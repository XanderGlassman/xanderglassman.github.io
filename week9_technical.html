Cross Site Scripting

Cross Site Scripting is kind of like SQLinjection except instead of ) and ;, the dangerous characters in this case are the angle brackets from HTML. If you don't "escape" angle brackets by converting those bracket into something that the browser doesn't see as an indicator of a tag (ie & l t ; [minus the spaces] for < and & g t ; for >), there will be a vulnerability in your site. This is because someone can put script tags into the input and then write whatever javascript they want. Apparently it's a pretty insidious problem, because anywhere a user inputs something there is the potential for that custom javascript to come into play. Some big companies even have bounties for exposing these weaknesses, because the other option, almost complete vulnerability, is so much less appealing.

SQL Injection

SQL is a the classic database language, around long before the Internet. However, because of its age, it is susceptible to SQL injection. Back in the day, you didn't have to worry about a user dropping some malicious input because you were the only one talking to the database. This left over issue means that a user can use dangerous characters like " and ; to end up creating a new command when inputing data into a database. This ultimately leaves all of the information in your database up for grabs, which is no good.

To avoid this you can use an escape string, which creates a function which puts a / in front of all the dangerous characters. The / tells the computer that the following character should only be interpreted only as text. This is usually a pretty good way to fix the problem, but there can be small bugs and whatnot. So another, more standard way to fix it to use prepared statements. These guys basically mark all the user's input as unsafe, telling the computer "Don't do anything beyond putting this stuff into the database, there are no commands here."

Cross Site Request Forgery (Week 9)

Today's web is apps rely on users to actively interact with web-pages though get requests. They're really great because they facilitate a user generated web, and a customized experience, but the problem is that they can be used by malicious website to mess with other web pages. For instance, if you logged into a web forum or somewhere with rather lax security, malicious websites can send a get request in the background to that website to say, delete your account on the other site. When the targeted site receives the request, it just executes the code, ignoring where it came from.

To fix this, HTML has a referrer header that tell a website where its request comes from. However, if users have some sort of customizing software in their browsers, as many do, the request is interpreted to be external and malicious. To avoid this while maintaining security, developers can use a nonce, aone time key that the user has to input in order to prove they're coming from the right spot.
